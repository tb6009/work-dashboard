# /월간회고 $ARGUMENTS

063_workDashboard의 월간 회고를 생성합니다.
**정량 분석(Claude 자동)** + **정성 코멘트(사용자 직접)** 2-레이어 구조로 `data/monthly/$ARGUMENTS.json`을 생성·갱신합니다.

---

## Step 0 — 입력 확인

1. `$ARGUMENTS` 형식이 `YYYY-MM`인지 확인 (아니면 중단하고 안내)
2. 해당 월의 주차 목록 계산 (ISO week 기준)
   - 예: 2026-05 → W18(04-27~), W19, W20, W21, W22(~05-31)
3. `app/src/data/weekly/` 에서 해당 주차 JSON 파일 전부 읽기
   - 없는 주차는 "데이터 없음"으로 표기하고 계속 진행
4. 기존 `app/src/data/monthly/$ARGUMENTS.json` 존재하면 읽어서 보존 (retro.userComment 덮어쓰지 않음)

---

## Step 1 — 정량 집계

### 1-A. 기본 KPI

weekly JSON의 `kpis` 합산:
- `totalFiles` = 각 주 `kpis.filesChanged` 합계
- `avgActive` = `kpis.activeProjects` 평균 (반올림)
- `newProjectsCount` = `kpis.newProjects` 합계
- `decisions` = 각 주 `decisions[]` 길이 합계
- `milestonesCount` = `decisions[].isMilestone === true` 건수 합계

### 1-B. 활동일 · Peak 계산

각 주 `daily[]` 전체를 순회:
- **activeDays** = 해당 월 날짜 중 `filesChanged > 0` 인 날 수
- **peakDay** = `filesChanged` 최댓값인 날 (date, weekday, files)
- **peakWeek** = 주 중 `kpis.filesChanged` 최댓값인 주

### 1-C. 프로젝트별 파일 집계

각 주 `projects[]` 순회해서 `projectFiles[id] += filesChanged` 누적.
`app/src/data/projects.json`에서 각 프로젝트 type 조회 → `typeBreakdown` 계산.

### 1-D. 토큰 비용

각 주 `daily[].entries[].tokens.costUSD` 합산 → 월간 총 costUSD, messages.

### 1-E. 마일스톤 목록

모든 주의 `decisions[]`에서 `isMilestone === true` 항목 추출 → 날짜순 정렬.

---

## Step 2 — 주차별 전수 분석 (누락 방지 핵심)

각 주를 분석할 때 **반드시 두 가지를 모두** 확인:

### A. `projects[]` — 공식 기여 목록
pct·filesChanged 기준 상위 프로젝트 파악.

### B. `daily[].entries[]` — **실제 작업 전수 목록** ← 이것이 핵심
entries의 `projectId`·`did` 전부 읽기.
**pct가 낮거나 projects[]에 없어도** entries에 의미 있는 작업이 있으면 반드시 narrative에 포함.

예시 — W21에서 092(영상과제)가 pct=35로 2위였지만 entries를 보면 4컷만화 대본·프롬프트 5종·샷리스트 등 파이프라인 전체가 이 주에 완성됐음. 이런 내용을 narrative에 넣어야 한다.

### 주차별 narrative 작성 기준:
- **수치가 낮아도 의미 있는 작업이면 반드시 포함**
- 산출물 이름을 구체적으로 (v0.1, v2.1, 3종 등)
- 해당 주를 관통하는 흐름/에너지/맥락을 한 문장으로 표현
- entries의 `[fs scan]` 항목은 파일명에서 작업 유추. `[ai]` 항목은 AI 보조 작업으로 표기
- 분량: 3~5문장

---

## Step 3 — 영역별 분포 narrative 작성

typeBreakdown의 각 영역에 대해:
1. 해당 영역 프로젝트들의 entries를 모아서 이달 주요 산출물 목록 파악
2. **단순 파일 수 나열 금지** — "무엇을 만들었고 어디까지 왔는가"를 서술
3. 분량: 2~4문장

---

## Step 4 — JSON 초안 작성

**파일**: `app/src/data/monthly/$ARGUMENTS.json`

기존 파일이 있으면:
- `aggregated`, `milestones`, `retro.stats`, `retro.weeklyFlow`, `retro.areaBreakdown`, `retro.keyMilestones` 갱신
- **`retro.userComment`는 기존 값 보존** (덮어쓰기 금지)

없으면 새로 생성. 스키마:

```json
{
  "month": "YYYY-MM",
  "weeks": ["YYYY-WXX", ...],
  "aggregated": {
    "totalFiles": N,
    "avgActive": N,
    "newProjectsCount": N,
    "milestonesCount": N,
    "typeBreakdown": { "product": N, "publishing": N, ... }
  },
  "topProjects": ["id1", "id2", ...],
  "milestones": [ ...모든 isMilestone=true 결정 ],
  "retro": {
    "headline": "N files · N active days · 한 줄 이달 성격",
    "stats": {
      "filesChanged": N,
      "activeDays": N,
      "totalDays": N,
      "decisions": N,
      "newProjectsRegistered": N,
      "peakDay": { "date": "YYYY-MM-DD", "weekday": "X", "files": N },
      "peakWeek": { "week": "YYYY-WXX", "files": N, "period": "MM-DD~MM-DD" },
      "costUSD": N,
      "messages": N
    },
    "weeklyFlow": [
      {
        "week": "WXX",
        "period": "MM-DD~MM-DD",
        "files": N,
        "label": "한 단어 성격",
        "headline": "핵심 성과 1줄",
        "narrative": "3~5문장 서술 — entries 전수 검토 후 작성"
      }
    ],
    "areaBreakdown": [
      {
        "area": "Product",
        "type": "product",
        "files": N,
        "pct": N,
        "topProjects": ["ID 이름 (Xf)", ...],
        "highlight": "한 줄 요약",
        "narrative": "2~4문장 서술"
      }
    ],
    "keyMilestones": [
      {
        "date": "MM-DD",
        "projectId": "XXX",
        "title": "마일스톤 제목",
        "impact": "의미와 파급효과 1~2줄"
      }
    ],
    "userComment": {
      "overall": "",
      "wentWell": ["", ""],
      "couldImprove": [""],
      "nextMonth": [""]
    }
  },
  "buildAt": "ISO timestamp"
}
```

keyMilestones는 전체 milestones 중 **영향력 기준 상위 5~7개** 선별. 선별 기준:
- 배포·제출·발표 등 외부 공개 산출물
- 방향 전환·폐기·재설계 등 중대 결정
- 인프라·체계 구축 중 이후 작업에 영향을 주는 것

---

## Step 5 — 초안 사용자 확인

JSON 작성 후, 다음 형식으로 내용을 채팅에 출력:

```
=== $ARGUMENTS 월간 회고 초안 ===

headline: [...]

주차별 흐름:
  WXX [레이블] Xf — headline
      narrative...

  WXX [레이블] Xf — headline
      narrative...

영역별 분포:
  Product XX% — highlight
      narrative...

핵심 마일스톤 N건:
  MM-DD [ID] title — impact

누락 가능성 체크:
  ⚠️ entries에는 있으나 projects[]에 없는 활동: [목록]
  ⚠️ pct가 낮지만 entries 기준 작업량이 많은 프로젝트: [목록]
```

출력 후 사용자에게 4개 질문을 **한 번에** 전달:

```
이 내용을 확인하셨으면 회고 코멘트 4가지를 답해주세요.

Q1. 이달 총평 — 한두 문장으로
Q2. 잘 된 것 — 2~3가지
Q3. 아쉬운 것 — 솔직하게
Q4. 다음 달 이어갈 것
```

## Step 6 — 사용자 코멘트 수신 후 최종 저장

사용자 답변을 받으면 `retro.userComment` 4개 필드를 채워서 JSON 갱신.
그대로 저장하되 너무 딱딱하면 자연스럽게 다듬어도 되나 반드시 사용자 확인 후 저장.

저장 완료 후:

```
✅ data/monthly/$ARGUMENTS.json 저장 완료
✅ retro.userComment 4개 필드 작성

▶ 다음 단계:
   git add app/src/data/monthly/$ARGUMENTS.json
   git commit -m "data: $ARGUMENTS 월간 회고 작성"
   git push
```

---

## 절대 규칙

1. **weekly JSON 수정 금지** — read-only
2. **기존 userComment 덮어쓰기 금지** — 빈 값일 때만 채움
3. **entries 전수 확인** — projects[] 상위만 보고 narrative 쓰지 말 것
4. **수치는 JSON에서 직접 계산** — 추정·어림 금지
5. **`data/projects/*.json` 이 커맨드에서 수정 금지** — 주간 업데이트에서 처리
