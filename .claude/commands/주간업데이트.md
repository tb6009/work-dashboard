# /주간업데이트 $ARGUMENTS

063_workDashboard의 주간 데이터를 빌드합니다. 이 명령은 `extract-week.mjs`가 생성한 draft JSON을 받아 의미적 필드를 채우고, 활동 있던 프로젝트 JSON을 갱신합니다.

---

## 입력 확인

먼저 다음을 확인하세요:

1. **Draft 존재 여부**: `_drafts/$ARGUMENTS_draft.json` 읽기
   - 없으면 사용자에게 `node scripts/extract-week.mjs $ARGUMENTS` 먼저 실행하라고 안내
2. **해당 주 raw 로그**: `../docs/logs/YYYYMMDD_log.md` (draft에 들어있는 날짜 범위)
3. **현재 weekly 상태**: `app/src/data/weekly/` 의 가장 최근 파일 (직전 주)
4. **프로젝트 메타**: `app/src/data/projects.json`

---

## 작업 절차

### Step 1 — Draft 읽고 raw 로그와 대조

Draft의 `_meta.todoFields`에 명시된 필드들을 채울 차례입니다.

**채워야 할 필드**:
- `summary` — 한 줄 핵심 요약 (예: "신규 3개 · 배포 1건 · 캐릭터 통합 1건")
- `projects[].imp` — 1~5 importance dot (5: 핵심 milestone, 1: minor)
- `projects[].nextAction` — 해당 프로젝트의 `PROJECT.md` "다음 액션" 1줄로 압축
- `decisions[].description` — draft는 title만 추출, raw 로그에서 1~2줄 설명 채움
- `decisions[].isMilestone` — 배포·중대 결정은 true
- `labelChanges` — `projects.json` status 비교로 검출

### Step 2 — 평가 기준

`pct` (이미 draft에 mechanical 추정 들어있음): 사용자가 주관적으로 조정 권장
`imp` 가이드:
- 5: 배포·외부 발표·논문 제출 등 milestone
- 4: 핵심 산출물 완성 (논문 섹션·앱 페이지·기획안 v1)
- 3: 일반 진척 (분석·설계·리서치)
- 2: 보완·정리 작업
- 1: 메타·관리 작업

### Step 3 — 최종 weekly JSON 작성

**저장 위치**: `app/src/data/weekly/$ARGUMENTS.json`

**스키마**: `app/src/types/dashboard.ts` 의 `WeeklySnapshot`. `_meta` 필드는 제거.

**검증**:
- 모든 `TODO` 문자열 제거되었는지
- `kpis.activeProjects` = `projects[]` 길이 일치
- `kpis.sessions` = `daily[]` 중 `filesChanged > 0` 일수
- `decisions[].projectId` 가 실제 활동 프로젝트인지

### Step 4 — 활동 있던 프로젝트 JSON 업데이트

`projects[]` 의 상위 5~10개 프로젝트에 대해:

**파일**: `app/src/data/projects/[id].json`

**갱신 필드**:
1. `currentSituation.summary` — 이번 주 종료 시점 상태로 갱신
2. `currentSituation.phase` — 단계 변경 있으면 갱신
3. `currentSituation.recentLogs` — 새 로그 1건 prepend (3건 초과 시 가장 오래된 것 제거)
   ```json
   {
     "date": "YYYY-MM-DD",
     "title": "이번 주 핵심 활동 1줄",
     "logFilePath": "/Users/jinhyunpark/Documents/cloude_Code/docs/logs/YYYYMMDD_log.md",
     "excerpt": "1-2줄 발췌 (Optional)"
   }
   ```
4. `deliverables` — 신규 산출물 추가 (이미 있으면 스킵). 절대경로 사용.
5. `nextTasks` — 완료된 항목 제거, 새 next action 추가
6. `currentSituation.blockers` — 신규 차단 요소 있으면 추가
7. `buildAt` — 현재 ISO 시각

### Step 5 — CURRENT_WEEK_ID 갱신

**파일**: `app/src/lib/data.ts`

```ts
export const CURRENT_WEEK_ID = '$ARGUMENTS';
```

이 한 줄만 교체. 다른 부분 손대지 말 것.

### Step 6 — 검증 출력

작업 완료 후 다음을 표 형태로 리포트:

```
✅ Weekly: app/src/data/weekly/$ARGUMENTS.json (kpis, projects N개)
✅ Projects updated: [id1, id2, ...]
✅ CURRENT_WEEK_ID: 2026-WXX
⚠️  Decisions: N건 / Milestones: M건
⚠️  Manual review needed: [pct 값 / 의미 모호한 결정 등]
```

마지막에:
```
▶ 다음 단계:
   cd app && npm run dev      # 시각 확인
   git diff app/src/data/      # 변경 확인
   git add . && git commit -m "weekly: $ARGUMENTS update" && git push
```

## 절대 규칙

1. **이전 주 weekly JSON 절대 수정 금지** — 영구 누적 원칙 (운영가이드 §9.6)
2. **`projects.json` 메타 변경 시 사용자 확인 필수** — status 라벨 자동 전환은 보수적으로
3. **존재하지 않는 파일 경로를 deliverables에 쓰지 말 것** — 의심되면 Read로 확인
4. **`_drafts/`는 보존** — 다음 주에 비교용 (덮어쓰기 OK, 삭제 금지)
5. **draft에 없는 데이터를 임의로 추가하지 말 것** — 출처 불명 데이터 차단

## 사용자 검토 트리거 (멈추고 물어볼 것)

- `pct` 합계가 100% 크게 초과 (>120%) 또는 미달 (<70%)
- 결정사항 중 milestone 후보가 5건 이상 (필터링 필요)
- 신규 프로젝트가 3건 이상 (라벨 정합성 확인 필요)
- 주간 활동 일수 0일 (휴무 주 처리)
