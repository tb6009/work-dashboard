3. 사용자가 폐기를 결정한 결정적 문제 7가지

3.1. 디테일 누락 (가장 큰 원인)

증상: 사용자가 "Week/Month의 정보 부분이 안 만들어진 거야?", "month의 데이터들이 보여주는 3가지 데이터 그래프도 독립적으로 유지" 지적.

원인: Plan 작성 시 "TodayTab/WeekTab/MonthTab → 각 ~120줄 분리"라는 라인 수 목표만 챙기고, 옮겨야 할 모든 영역을 명세하지 않음.

  • Plan 2 Task 5(WeekTab 슬림화)에는 WeekChart + WeekReview만 명시 → 기존 7~9개 정보 영역 누락
  • Plan 2 Task 7(MonthTab 슬림화)에는 캘린더 + MonthReview만 명시 → 기존 5개 통계·차트·AI Report 누락

책임: Plan 작성자(저). Plan 5(Diary Stats Restore)로 후속 복원했으나 한 라운드 추가 + 디테일 손실 잔재.

3.2. DateSelector 인터랙션 재작성

증상: 사용자가 "지금처럼 날짜 탭이 스크롤되는게 어딨어??? 이런 아님."

원인: 1203e05 원본 패턴은 가운데 박스 고정 + 날짜들이 좌우 스와이프로 들어옴 (scroll handler + centeredIndex). 새 DateSelector(Plan 2)는 scrollIntoView + opacity fade만으로 시도 — 패턴 자체가 다름.

책임: SPEC §2.2 Today selector 항목에 "정중앙 정렬 + 양옆 fade"만 적었지 기존 인터랙션 패턴(scroll handler 기반 picker)을 명시 안 함. 시안에서는 정적 이미지로 표현 → 인터랙션 차이가 시안 검토로 잡히지 않음.

3.3. /haru 분리 시 작동 불가

증상: "하루도 지금 작동을 안 해요. 또 하루의 입력 창도 위치가 안 맞습니다."

원인 1 (자동 조정 잔재): Plan 3 subagent가 useChatMessages hook 만들 때 saveChatMessage 인자 순서를 실제 api 시그니처에 맞춤 + DB message 컬럼 → text 매핑 추가. 합리적 자동 조정이지만 1203e05 원본의 직접 호출 패턴과 다르게 작동.

원인 2 (입력창 위치): 1203e05 입력창은 column flex의 마지막 element + flexShrink: 0. 분리 버전(ChatInput)도 같은 패턴이지만 layout.tsx의 paddingBottom: 70px와 만나면 미세한 차이 발생. iPhone 13 safe-area-inset-bottom 고려가 명시되지 않음.

원인 3 (별개 환경 이슈): .env.local에 OPENAI_API_KEY 없음 + Vercel env에도 없음. 사용자가 production에서 작동했던 건 localStorage 키 덕분. v2.0과 무관한 환경 설정 문제지만 같은 시기에 발견되어 v2.0이 원인처럼 보임.

3.4. dev 서버 브랜치 혼동

증상: 사용자가 localhost:3001에서 "안 됨"이라 보고 → 디버깅 시간 낭비.

원인:

  • app/ (main 브랜치) — 1203e05 복원판 + fix들 push
  • app-stage9/ worktree (plan-5 브랜치) — 분리 버전, /haru 작동 안 함
  • dev 서버는 app-stage9 worktree에서 시작 (port 3001)
  • 사용자는 localhost:3001 보고 있었음 → plan-5 브랜치 코드

같은 commit이 떠 있다고 가정하고 진단 → "왜 안 되지?" 한참 헤맴. 환경 격리를 처음부터 명확히 안 알림.

3.5. 시안 검토만으로 인터랙션 검증 불가

증상: 시안 17_final-design-spec.html은 OK 받음 → 코드 구현 후 작동 차이 발견.

원인: 시안은 정적 mockup. 가운데 박스 고정 vs scrollIntoView는 그림으로는 같아 보임 — 실제 만져봐야 차이 인지. 시안 OK = 디자인 OK이지 인터랙션 OK가 아님.

3.6. Plan 대비 자동 조정 = 미세한 동작 차이

증상: subagent 보고에 "Plan 대비 자동 조정 5건" — 합리적으로 보이는 보고서지만 그 자동 조정들이 사용자 체감 차이를 만듦.

예시:

  • getYearWeek 공식 — plan 안에서 자기모순 → 테스트 기준으로 수정 (잘한 일)
  • vitest.config.ts → .mts — ESM 호환 (필요한 일)
  • useChatMessages race condition fix — 작동에 필요 (그러나 1203e05엔 없던 race가 분리로 인해 생긴 것)
  • vi.stubEnv 사용 — Node v22 호환 (테스트 인프라)

자동 조정 자체는 합리적이지만, 누적되면 "원본과 미세하게 다른 동작"이 됨. 각 조정은 plan 명시 허용 범위 안이라 사용자 검토 게이트를 통과.

3.7. 분리 = 좋다는 가정의 함정

증상: SPEC 영역 5(아키텍처)에서 "큰 파일 4개 분리"를 P0(반드시)로 분류. 사용자는 "P0+P1 모두 채택"으로 동의.

원인: 사용자가 "분리해도 사용성·인터랙션 100% 동일"이라고 믿고 OK. Claude도 그렇게 약속함. 그러나 실제로는:

  • 분리하면서 부모-자식 props 인터페이스를 설계 → 일부 미세 상태가 사라짐
  • 인라인 핸들러를 추출하면서 컨텍스트(예: parent state) 의존성이 props로 변환 → 의도 약간 변경
  • 단일 책임 원칙은 좋은 패턴이지만 이미 작동하는 코드에 적용할 때 항상 회귀 위험

SPEC §6 "기존 사용자 데이터 무손상" 보장은 했지만 "사용자 체감 동작 무손상" 보장은 명시하지 않음.

6. 정리 / 향후 방지책

6.1. 사용자 의도 해석 시

  • "개선해 주세요"는 새 기능 추가의 의미. refactor 동의는 별도 명시 받기.
  • "사용성 100% 유지 + 새 기능 layer 추가" 형태가 가장 안전.
  • 분리(refactor)는 사용자가 명시 요청할 때만 진행.

6.2. Plan 작성 시

  • 큰 파일 분리할 때는 옮겨야 할 모든 영역을 명시적 inventory로 정리. "~120줄" 같은 라인 수 목표는 부차적.
  • 인터랙션 패턴(scroll handler, hover/touch state, keyboard 처리 등)은 시각 mockup으로는 검증 안 됨. 별도 인터랙션 명세 필요.

6.3. 검증 게이트

  • Vercel preview URL ≠ 충분한 검증. 특히 인터랙션 디테일.
  • main 머지 전에 dev 서버에서 사용자가 충분히 만져본 다음에만 머지.
  • 큰 변경은 단일 머지보다 점진 머지 (작은 PR 여러 개) 권장 — 회귀 발견 즉시 차단 가능.

6.4. 환경 격리 안내

  • worktree 사용 시 각 환경(dev 서버 포트 등)이 어떤 브랜치 코드를 보는지 매 turn마다 명시.
  • 사용자에게 "지금 보고 계신 게 어느 브랜치 코드인지" 확인을 도와야.

6.5. 롤백 안전망

  • 첫날부터 "사용자 만족 못 하면 즉시 1203e05로 복원 가능" 명시했어야.
  • force push는 안전 가드로 차단됨 → git read-tree --reset + 단일 commit 패턴이 가장 안전한 롤백 방식.

6.6. 환경 설정 별개 이슈 분리

  • .env.local / Vercel env에 OPENAI_API_KEY 부재는 1203e05부터 있던 환경 문제 — 작업 시작 시 점검했어야.
  • 작업 도중 발견된 별개 이슈는 별도 항목으로 분리해 사용자에게 보고. 작업 결과로 오해 안 받게.

8. OpenAI API Key 별도 이슈 (v2.0과 무관)

작업 중 발견됐지만 v2.0과 무관한 환경 설정 문제. 정리:


# .env.local 현재 상태
ANTHROPIC_API_KEY=...
GOOGLE_CLIENT_ID=...
GOOGLE_CLIENT_SECRET=...
NEXTAUTH_SECRET=...
NEXTAUTH_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
NEXT_PUBLIC_SUPABASE_URL=...
# OPENAI_API_KEY=...  ← 빠짐

/haru는 localStorage("openai_api_key") 또는 process.env.OPENAI_API_KEY 필요. 현재 둘 다 없으면 작동 안 함. 1203e05 / v0.8.1 시점에도 동일한 상태였음.

조치 권장:

  • 로컬 dev: /settings에서 사용자 key 입력 또는 .env.local에 추가
  • production: Vercel 대시보드 → Settings → Environment Variables → OPENAI_API_KEY 추가

9. 결론

v2.0이 폐기된 가장 큰 이유는 사용자 의도 해석 미스매치 + 분리 작업의 미세 부작용입니다. 새 기능 자체(회고 색상 시스템, 시간 네비 등)는 가치 있었지만, 그걸 위해 동반된 거대 refactor가 익숙한 UI 디테일을 미세하게 깨뜨렸고, 사용자 입장에서는 "안 된다"는 종합 체감으로 누적됐습니다.

기술적으로는 모든 작업이 plan에 따라 진행됐고 테스트도 통과했지만, 사용자 검증이 최종 판단이며 그 판단이 부정적일 때 즉시 롤백할 수 있는 안전망이 가장 중요합니다.

v0.8.1로 안전하게 복원된 현재가 적절한 결정입니다. 향후 새 작업 시 본 리포트의 6절 방지책을 적용하겠습니다.
