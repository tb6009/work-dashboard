## 폐기의 결정적 원인 7가지

| # | 원인 | 심각도 |
|---|------|--------|
| 1 | **디테일 누락** — Plan에 옮겨야 할 영역 명세 빠짐 | Critical |
| 2 | **DateSelector 인터랙션 재작성** — scroll handler picker → scrollIntoView | Critical |
| 3 | **/haru 작동 불가** — hook 분리 시 인자 순서 변경 + 환경변수 부재 | Critical |
| 4 | **dev 서버 브랜치 혼동** — worktree 환경 안내 미흡 | High |
| 5 | **시안 = 인터랙션 검증 불가** — 정적 mockup OK ≠ 동작 OK | High |
| 6 | **자동 조정 누적** — 각각은 합리적이지만 합치면 원본과 다름 | Medium |
| 7 | **"분리 = 좋다" 가정** — 작동하는 코드의 refactor는 항상 회귀 위험 | Root Cause |

## 핵심 교훈

> **사용자가 원한 것**: 디테일·인터랙션 100% 유지 + 새 기능을 layer로 얹기
> **실제 진행된 것**: 큰 파일 분리(refactor) + 새 기능 + 일부 디테일 손실

### 향후 원칙

1. **"개선해 주세요"는 새 기능 추가의 의미. refactor 동의는 별도 명시.**
2. **기존 인터랙션 패턴은 코드 수준으로 명세** (시안으로는 검증 불가)
3. **main 머지 전 사용자가 dev에서 충분히 사용 후 승인**
4. **큰 변경은 점진 머지** (작은 PR 여러 개)
5. **즉시 롤백 가능한 안전 commit 항상 확보**
