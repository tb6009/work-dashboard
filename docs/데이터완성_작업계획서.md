# 063 workDashboard — 데이터 완성 작업계획서

> **작성**: 2026-05-06
> **목적**: 현재 폴더에 있는 데이터(20개 프로젝트 JSON · W18 weekly 1개)를 기준으로 모든 페이지가 제대로 작동하도록 완성
> **승인 후 실행**: 본 문서 검토 → 사용자 승인 → 단계별 실행

---

## 0. 작업 범위 정의

**"현재 있는 데이터" 의 정확한 범위**:
- ✅ `app/src/data/projects.json` — 22개 프로젝트 메타
- ✅ `app/src/data/projects/*.json` — 20개 detail (00 제외)
- ✅ `app/src/data/weekly/2026-W18.json` — 1개 weekly
- ✅ `../docs/logs/*.md` — 11개 일일 로그 (3/29 ~ 4/27, raw)
- ❌ monthly · yearly 집계 — 0개

**완성의 정의**:
- 5개 페이지(`/`, `/weekly`, `/monthly/[month]`, `/yearly/[year]`, `/projects/[id]`) 모두 하드코딩 제거
- 모든 페이지가 `data/` 폴더의 실제 JSON 파일에서 데이터를 읽어 렌더링
- 데이터 없는 기간(미래·미수집 주)은 graceful "데이터 미수집" UI로 처리
- 신규 데이터 추가 시 코드 수정 없이 자동 반영되는 구조

## 2. 작업 단계 — 3 Phase

### Phase 1 — 데이터 로더 확장 (`lib/data.ts`)

**소요**: 약 20분
**위험도**: 낮음 (신규 함수만 추가)

신규 추가 함수:
```typescript
// 모든 weekly JSON 파일 스캔 → 정렬된 배열 반환
loadAllWeeks(): Promise<WeeklySnapshot[]>

// 특정 월의 weekly들을 모아 monthly 형태로 집계
loadMonth(monthId: string): Promise<MonthlySnapshot | null>
//   → data/monthly/{monthId}.json 우선 시도
//   → 없으면 data/weekly/*.json에서 동적 집계 (fallback)

// 특정 연도의 monthly들을 모아 yearly로 집계
loadYear(yearStr: string): Promise<YearlySnapshot | null>
//   → data/yearly/{yearStr}.json 우선 시도
//   → 없으면 monthly에서 동적 집계 (fallback)

// 모든 weekly id 리스트 (정렬됨, 중복 제거)
listWeekIds(): Promise<string[]>
```

**구현 전략**:
- Next.js 16 정적 라우트는 빌드 타임 fs 스캔 필요 → `import.meta.glob` 또는 `fs/promises` + 파일명 패턴
- 동적 집계는 weekly 데이터 합산 (totalFiles = sum, decisions = concat 등)

**결과물**: `lib/data.ts` 36줄 → 약 130줄

---

### Phase 2 — 페이지 리팩토링 (4개 페이지)

**소요**: 약 70분
**위험도**: 중간 (UI는 그대로 유지, 데이터 소스만 교체)

#### 2-1. `/` 홈 페이지 (10분)

- `RECENT_WEEKS` 하드코딩 제거
- `loadAllWeeks()` 호출 → 최근 5주 추출
- 데이터 없는 주는 "데이터 미수집" 카드

**유지할 것**:
- 헤로 영역, KPI, daily activity bar 등 UI 컴포넌트는 그대로
- W18 데이터 표시 로직 그대로

#### 2-2. `/weekly` 페이지 (25분) — 가장 큰 변경

- `WEEKLY_DATA` 18주 하드코딩 제거 → `loadAllWeeks()` 사용
- `MONTHS` 12개월 정의는 유지 (이건 정적 캘린더 메타데이터)
- `MonthSection` 안에서 각 주를 weekly 데이터와 매칭
- **카드를 실제 링크로**: `href="#"` → `/projects/[topId]` 또는 향후 `/weekly/[weekId]` 상세 페이지
- 데이터 없는 주는 회색 placeholder (현재 `future` 처리와 동일)

**중요 결정**: 카드 클릭 시 어디로 갈지?
- 옵션 A: 그 주의 톱 프로젝트 디테일로 (간단)
- 옵션 B: `/weekly/[weekId]` 상세 페이지 신설 (Phase 3로 미룸)
- **권장**: 옵션 A — 지금은 톱 프로젝트로 보내고, 나중에 상세 페이지 만들면 그때 라우트만 변경

#### 2-3. `/monthly/[month]` 페이지 (20분)

- `WEEK_CARDS_2026_05`, `MILESTONES_2026_05` 하드코딩 제거
- `loadMonth(month)` 사용
- `params.month` 받아서 동적으로 어떤 달이든 렌더링
- W18만 있으므로 5월 → W18 1주만 표시, 4월·3월 → "데이터 미수집" 상태

#### 2-4. `/yearly/[year]` 페이지 (15분)

- `MONTH_CELLS`, `COMPOSITION`, `GANTT_ROWS` 하드코딩 제거
- `loadYear(year)` 사용
- 5월(W18 데이터)만 채워지고 나머지는 빈 상태

**유지할 것**:
- 차트 컴포넌트 (`TypeStackedArea`, `ProjectGantt`) 그대로
- 입력 데이터만 동적 집계 결과로 교체

### Phase 3 — 보조 인프라 (선택)

**소요**: 약 30분
**위험도**: 낮음 (신규 파일)

#### 3-1. 월간·연간 집계 스크립트

`scripts/aggregate-month.mjs`:
- 입력: `2026-05`
- 처리: `data/weekly/` 중 해당 월에 속한 모든 JSON 합산
- 출력: `data/monthly/2026-05.json`
- 용도: 월말 자동 생성, Phase 4 자동화 단계 준비

`scripts/aggregate-year.mjs`:
- 동일 패턴으로 yearly 집계

**현재 가치**:
- 페이지는 Phase 1의 동적 집계로 작동하므로 이 스크립트들은 **선택**
- 하지만 빌드 타임 성능 최적화 + Phase 4 cron 준비를 위해 미리 만들어두면 좋음

#### 3-2. 가이드 문서 갱신

`docs/주간업데이트_가이드.md`에 다음 추가:
- 월말에 `aggregate-month.mjs` 실행 권장
- 연말에 `aggregate-year.mjs` 실행 권장

## 3. 작업 순서·체크포인트

```
[Phase 1] lib/data.ts 확장
    ↓ 체크포인트: 함수 단위 테스트 (간단한 console.log 호출)
[Phase 2-1] 홈 / 페이지
    ↓ 체크포인트: dev 서버 재시작, http://localhost:3001/ 시각 확인
[Phase 2-2] /weekly 페이지
    ↓ 체크포인트: /weekly 시각 확인, 카드 클릭 작동 확인
[Phase 2-3] /monthly/[month] 페이지
    ↓ 체크포인트: /monthly/2026-05, /monthly/2026-04 둘 다 확인
[Phase 2-4] /yearly/[year] 페이지
    ↓ 체크포인트: /yearly/2026 확인
[Phase 3] 집계 스크립트 (선택)
    ↓ 체크포인트: aggregate-month.mjs 실행 → JSON 생성 검증
[전체 검증] npm run build 성공 확인
```

각 체크포인트마다 사용자 확인 후 다음 Phase 진행. 자동 모드여도 페이지 작동 검증은 사용자 시각 확인이 필요한 영역.

## 4. 위험·트레이드오프

### 위험 1: Next.js 16의 정적 라우트 + fs 스캔
- `import.meta.glob` (Vite) 또는 `fs.readdir` (Node.js) 사용 시 dev/build 동작 차이
- **완화**: 최소 한 번은 `npm run build` 로컬 검증 필수

### 위험 2: 하드코딩 데이터 제거 시 시각적 빈곤
- W18 외에 데이터 없는 주는 모두 "데이터 미수집" 회색 카드
- 사용자가 보기에 "기능 후퇴"로 느껴질 수 있음
- **완화**: 빈 상태 디자인을 명확히 ("아직 수집 안 됨" 라벨, 향후 채워질 거란 시그널)

### 위험 3: 4개 페이지 동시 변경 시 회귀 가능성
- **완화**: Phase 2를 1-2-3-4 순차 진행, 각 페이지 후 시각 확인 체크포인트

### 트레이드오프: 데이터 페이지 vs 백필
- **본 계획**: 페이지 리팩토링만 (백필 없음)
- 결과: 페이지는 깨끗하게 작동하지만 W18 외 모든 주는 빈 상태
- 백필 추가하면 풍성해지지만 별도 작업 (다음 세션 권장)
